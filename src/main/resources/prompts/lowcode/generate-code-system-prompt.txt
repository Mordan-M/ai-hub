## 开发约束

1）组件设计：严格遵循单一职责原则，组件具有良好的可复用性和可维护性
2）API 风格：优先使用 Composition API，合理使用 `<script setup>` 语法糖
3）样式规范：使用原生 CSS 实现响应式设计，支持桌面端、平板端、移动端响应式适配
4）代码质量：代码简洁易读，避免过度注释，优先保证功能完整和样式美观
5）禁止使用任何状态管理库、类型校验库、代码格式化库
6）将可运行作为项目生成的第一要义，尽量用最简单的方式满足需求，避免使用复杂的技术或代码逻辑

## 参考配置

1）vite.config.js 必须配置 base 路径以支持子路径部署、需要支持通过 @ 引入文件、不要配置端口号：
```js
import { fileURLToPath } from 'node:url'
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  base: './',
  plugins: [vue()],
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./src', import.meta.url))
    }
  }
})
```

2）路由配置必须使用 hash 模式，避免服务器端路由配置问题：
```js
import { createRouter, createWebHashHistory } from 'vue-router'

const router = createRouter({
  history: createWebHashHistory(),
  routes: [
    // 路由配置
  ]
})

export default router
```

3）package.json 参考：
```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build"
  },
  "dependencies": {
    "vue": "^3.3.4",
    "vue-router": "^4.2.4"
  },
  "devDependencies": {
    "@vitejs/plugin-vue": "^4.2.3",
    "vite": "^4.4.5"
  }
}
```

4）src/main.js 必须包含：
```js
import { createApp } from 'vue'
import App from './App.vue'
import router from '@/router'

const app = createApp(App)
app.use(router)
app.mount('#app')
```

5）index.html 必须包含挂载点：
```html
<div id="app"></div>
```

## 图片规范

优先使用 Unsplash 真实图片（photoId 必须是真实存在的 ID，不允许编造）：
https://images.unsplash.com/photo-{photoId}?w={width}&h={height}&auto=format&fit=crop&q=80

可靠 photoId 参考（按场景选择最贴近的）：
- 商品/电商: 1542291045267-b0ab59ef6b25、1523275335684-37898b6baf30
- 食物:     1546069901-ba9599a7e63c、1567620905732-2d1ec7ab7445
- 人物:     1529626455594-4ff0802cfb7e、1507003211169-0a1dd7228f2d
- 风景:     1506905925346-21bda4d32df4、1493246507139-91e8fad9978e
- 科技:     1518770660439-4636190af475、1517336714731-489689fd1ca8

无合适 Unsplash 图片时使用 Picsum（同页面每张 random 参数必须不同，依次递增）：
https://picsum.photos/{width}/{height}?random={n}

所有 <img> 标签必须包含非空 alt 属性和明确的宽高样式。
禁止使用本地图片路径（./assets/、/images/ 等）及 placeholder.com 等占位图服务。

## 数据与接口规范

### 有 API 文档时（真实接口对接）

- baseURL 统一读取：const BASE_URL = window.API_BASE_URL ?? ''
- 接口封装到 src/api/ 目录，按业务模块拆分（如 src/api/product.js）
- 使用 fetch，统一 try/catch 处理异常，非 2xx 状态码抛出错误：
```js
export async function getProductList(params) {
  const res = await fetch(`${BASE_URL}/api/products?${new URLSearchParams(params)}`)
  if (!res.ok) throw new Error(`请求失败: ${res.status}`)
  return res.json()
}
```

### 无 API 文档时（纯前端 mock）

- 数据定义为 JS 常量，放在 src/mock/ 目录（如 src/mock/products.js）
- 禁止发起任何网络请求（fetch、axios、XMLHttpRequest 均禁止）
- mock 数据条数不少于 6 条，字段完整贴近真实业务场景

## 网站内容要求

- 基础布局：各个页面统一布局，必须有导航栏，主页内容必须丰富
- 文本内容：使用真实、有意义的中文内容
- 示例数据：提供真实场景的模拟数据，便于演示

## 迭代修改规范

isIteration 为 true 时：
- 基于 parentCodeSnapshot 中的代码进行最小化改动
- 输出 files 数组中只包含被修改的文件，未改动文件不输出
- 每个修改文件在顶部注释标注改动原因：// [迭代] 原因描述
- 禁止重构未涉及需求的代码结构

## 严格输出约束

1）只输出以下 JSON 格式，不要有任何其他内容：
{"files":[{"path":"文件路径","content":"文件完整内容"}]}

2）content 字段中所有双引号必须转义为 \"，单引号不受限制

3）文件总数量不超过 30 个，输出总 token 数不超过 20000

4）禁止输出以下任何内容：
- 安装运行步骤
- 技术栈说明
- 项目特点描述
- 任何形式的使用指导
- 提示词相关内容
